# Customer Churn Prediction using Decision Tree Classifier

An interpretable machine learning project that predicts customer churn for a telecommunications company using an **Entropy-based Decision Tree**, with **RandomizedSearchCV** hyperparameter tuning built on scikit-learn.

---

## Dataset

**Source:** [Telco Customer Churn — Kaggle](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)

The dataset contains **7,043 customer records** across **21 features** covering demographics, subscribed services, account details, and a binary churn label (`Yes` / `No`).

| Feature Category | Examples |
|---|---|
| Demographics | `gender`, `SeniorCitizen`, `Partner`, `Dependents` |
| Services | `PhoneService`, `InternetService`, `StreamingTV`, `OnlineSecurity`, `TechSupport` |
| Account Info | `Contract`, `PaymentMethod`, `MonthlyCharges`, `TotalCharges`, `tenure` |
| Target | `Churn` (Yes / No) |

---

## Project Workflow

### 1. Environment Setup
- Standard libraries imported: `numpy`, `pandas`, `matplotlib`, `seaborn`, `scikit-learn`.
- `DecisionTreeClassifier`, `compute_class_weight`, and `sklearn.tree` imported for model building and visualization.
- Random seed fixed at `SEED = 42` for full reproducibility.
- Dataset loaded from Google Drive via Google Colab.

### 2. Exploratory Data Analysis (EDA)

A thorough EDA is performed before any modelling:

- **Shape & Info:** 7,043 rows × 21 columns; mixed dtypes (float64, int64, object).
- **Missing Values:** Zero null entries confirmed.
- **Duplicates:** No duplicate rows found.
- **Numerical Stats:** Range, mean, median, and standard deviation profiled for `tenure` (0–72 months), `MonthlyCharges` ($18.25–$118.75), and `SeniorCitizen` (binary, 16.21% seniors).
- **Churn Distribution:** Clear class imbalance identified — **73.46% No Churn vs 26.54% Churn**, with a 2.77:1 No:Yes ratio.
- **Visualizations:** Bar chart of churn distribution, plus additional feature-level plots to understand behavioral and demographic patterns influencing churn.

### 3. Data Preprocessing

| Step | Details |
|---|---|
| Drop ID column | `customerID` removed — not a predictive feature |
| Target encoding | `Churn` mapped from `Yes/No` → `1/0` |
| Feature/target split | `X` = all features (19 columns), `y` = Churn label |
| One-hot encoding | `pd.get_dummies(drop_first=True)` applied to categorical columns, expanding to **19 features** |
| Imputation | `SimpleImputer(strategy='mean')` applied to handle any residual NaN values post-encoding |
| Train/Test Split | 80/20 split → **5,634 training** / **1,409 test** samples |
| Label encoding | `LabelEncoder` applied to any remaining object-type columns in train/test sets |
| Feature scaling | `StandardScaler` applied — fit on training data, transformed on both sets |

> **Note:** Unlike SMOTE-based approaches, this notebook addresses class imbalance using the `class_weight` parameter during hyperparameter search rather than resampling.

### 4. Baseline Decision Tree Model

An initial Decision Tree is trained to establish a performance baseline before tuning:

```python
DecisionTreeClassifier(
    criterion="entropy",   # Information Gain (entropy) as split criterion
    max_depth=3,           # Shallow tree to prevent overfitting
    min_samples_leaf=5,    # At least 5 samples required per leaf
    random_state=42
)
```

**Baseline Results:**

| Metric | Score |
|---|---|
| **Accuracy** | **78.57%** |
| Precision (Churn) | 0.67 |
| Recall (Churn) | 0.36 |
| F1 Score (Churn) | 0.46 |

The baseline model achieves decent accuracy but has low recall on the churn class — prompting the need for hyperparameter tuning.

### 5. Hyperparameter Tuning with RandomizedSearchCV

A wide search space is explored to find the optimal Decision Tree configuration:

```python
param_grid = {
    'criterion':          ['gini', 'entropy'],
    'max_depth':          [None, 10, 20, 30, 40, 50],
    'min_samples_split':  [2, 5, 10, 20, 30],
    'min_samples_leaf':   [1, 2, 5, 10, 15, 20],
    'max_features':       [None, 'sqrt', 'log2'],
    'class_weight':       [None, 'balanced']
}
```

`RandomizedSearchCV` is used with:
- **100 random candidates** sampled from the grid
- **5-fold cross-validation** (totalling 500 fits)
- Scoring metric: `accuracy`
- `n_jobs=-1` for parallel execution

The best estimator is extracted and evaluated on the held-out test set.

### 6. Tuned Model Results

**Post-Tuning Performance:**

| Metric | Score |
|---|---|
| **Accuracy** | **80.06%** |
| Precision (Churn) | 0.65 |
| Recall (Churn) | 0.51 |
| F1 Score (Churn) | 0.57 |
| **ROC AUC** | **0.7052** |

Tuning improved Recall on the churn class from **0.36 → 0.51** — a significant gain in catching actual churners — while keeping overall accuracy above 80%.

### 7. Visualizations

- **Decision Tree Plot:** The tuned tree is plotted using `sklearn.tree.plot_tree` (limited to 3 levels for readability), with color-filled nodes indicating class predictions, enabling human-interpretable decision paths.
- **Confusion Matrix Heatmap:** Seaborn heatmap showing true/false positives and negatives across No Churn / Churn classes.
- **ROC Curve:** AUC curve plotted for both the baseline and tuned models to visualize classifier performance across all classification thresholds.

## Where to Use This Model

The interpretability of a Decision Tree makes it particularly valuable in settings where **explainability matters as much as performance**:

- **Telecom & ISP Companies** — Identify customers likely to cancel their subscriptions and feed those predictions into targeted retention campaigns.
- **SaaS & Subscription Businesses** — Flag users approaching churn before they cancel, enabling proactive engagement by customer success teams.
- **Business Intelligence Dashboards** — Embed churn scores into BI tools (Tableau, Power BI) to give sales and support teams real-time at-risk customer views.
- **CRM Integration** — Push churn probability scores directly into CRM platforms (Salesforce, HubSpot) to trigger automated win-back workflows or discount offers.
- **Regulatory & Compliance Settings** — Unlike black-box models, the Decision Tree's visual decision paths can be presented to stakeholders or auditors to justify model-based decisions.
- **Rapid Prototyping** — As a lightweight, fast-training model, this is ideal for quickly validating whether churn prediction is viable on a new dataset before investing in more complex architectures.
- **Marketing Segmentation** — Segment customers by churn risk level and tailor messaging — for example, offering loyalty discounts to high-risk month-to-month contract holders.

## Future Enhancements

| Enhancement | Description |
|---|---|
| **SMOTE / Class Resampling** | Add SMOTE or ADASYN oversampling in a pipeline to further improve Recall on the minority churn class, complementing the current `class_weight='balanced'` approach. |
| **Ensemble Upgrade** | Extend to Random Forest, Gradient Boosting (XGBoost, LightGBM), or AdaBoost for higher predictive accuracy while retaining some interpretability via feature importance. |
| **SHAP Explanations** | Integrate SHAP values to generate per-customer explanations — e.g. "This customer is likely to churn because of their month-to-month contract and high monthly charges." |
| **Feature Selection** | Apply Recursive Feature Elimination (RFE) or mutual information scoring to reduce dimensionality and identify the minimal feature set needed for accurate predictions. |
| **Threshold Optimization** | Tune the classification threshold beyond the default 0.5 to maximize F1 or Recall depending on the business cost of false negatives (missed churners) vs. false positives. |
| **Stratified Cross-Validation** | Use `StratifiedKFold` during tuning to ensure the minority churn class is consistently represented in every fold, producing more reliable CV scores. |
| **GridSearchCV for Final Tuning** | After narrowing the search space with RandomizedSearchCV, apply a focused `GridSearchCV` for exhaustive fine-tuning of the top candidate configurations. |
| **Pipeline Integration** | Wrap preprocessing (imputation, scaling, encoding) and the model into a single `sklearn.Pipeline` for cleaner, reproducible, and deployment-ready code. |
| **Temporal Churn Indicators** | Enrich features with rolling behavioral metrics (e.g., usage trends, support ticket frequency) for more accurate predictions on time-series customer data. |

## Scalability Considerations

| Area | Strategy |
|---|---|
| **Large Datasets** | scikit-learn's `DecisionTreeClassifier` is efficient for tabular data; for datasets beyond a few million rows, consider `LightGBM` or `Dask-ML` for distributed training. |
| **Model Serialization** | Export the trained model with `joblib.dump()` for fast, lightweight deployment — Decision Trees serialize and load far more quickly than neural networks. |
| **Batch Inference** | Run nightly/weekly batch scoring of the entire customer base using cloud functions (AWS Lambda, GCP Cloud Functions) or scheduled Airflow DAGs. |
| **REST API Serving** | Wrap the serialized model in a FastAPI or Flask endpoint to serve real-time churn probabilities as customers interact with the product. |
| **Retraining Pipeline** | Automate periodic retraining with fresh data using MLflow or Kubeflow Pipelines, tracking model performance over time to detect degradation. |
| **Containerization** | Dockerize the training and inference environment for consistent, platform-independent deployments across development, staging, and production. |
| **Monitoring & Drift Detection** | Integrate Evidently AI or WhyLabs to detect feature drift and alert when incoming customer distributions diverge significantly from training data distributions. |
